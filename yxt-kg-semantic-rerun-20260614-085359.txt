欢迎收听这期技术小记。

今天我们收尾的是一条从设计修正到真实评测的链路。主题很明确：知识图谱不应该再被简单地塞进 RAG 的排序结果里，而应该作为 Agent 可以主动调用的工具。

一开始的问题，是旧的 KG 评测流程其实已经不成立了。以前的 hybrid-kg 和 mixed，会把普通文本 chunk 和图谱结果混在一起，再做一次排序。这个做法可以做消融，但它没有给 Agent 一个真正的图谱使用方式。尤其是它没有语义入口，Agent 不能先通过语义搜索找到候选实体，再用 Cypher 展开关系。因此，那份旧评测不能代表现在的能力。

这次我们把流程重新摆正了。文本检索继续走 Omni Search 的 hybrid，也就是向量加 BM25。知识图谱这边，Agent 有两个工具：kg-semantic 用 LadybugDB 内置的真实向量搜索找实体；kg-cypher 用来查节点、关系、邻居和路径。这样 Agent 可以先查原文，再查图谱线索，最后回到原文核对引用。

为了让评测真的跑在新能力上，glmcc 的图数据库也重新确认过。KgEntity 表已经带 embedding DOUBLE 数组，节点是一千八百六十七个，关系是三千六百四十三条。语义查询“动态用户组”可以命中动态用户组、删除动态用户组、创建动态用户组等实体；普通 Cypher 查询也能查到动态用户组和培训项目、考试、客户经理等关系。

接下来是新的评测。我们让 GLMCC 和 MiniMaxCC 两个 answer agent 各自跑完整的一百五十二题，每边并发三路。为了避免两个任务同时写同一个结果目录，还给 rag-eval 补了一个 generate-only 模式：先并行生成答案，只写各自 answers 目录；再分别用 from-answers 做 judge 和 score。这个小改已经提交到 rag-pipeline，提交号是 46cc911。

结果也已经落库并推送到 rag-benchmark。GLMCC 这组一百五十二题全生成，正确一百二十六题，部分正确十七题，错误九题，加权准确率八十八个百分点。MiniMaxCC 这组同样一百五十二题全生成，正确一百一十七题，部分正确二十六题，错误九题，加权准确率八十六个百分点。对应的结果目录是 eval semantic rerun glmcc v1 和 eval semantic rerun minimaxcc v1，汇总文档是 semantic rerun kg agent。

最后，旧评测已经在文档里标记为 legacy ablation，只保留作历史消融，不再拿来判断 KG Agent 的效果。现在这条线的含义更清楚：文本负责证据，知识图谱负责结构，语义搜索负责入口，Agent 负责决定什么时候继续查。

这就是今天的故事。我们把一个“看起来分数还可以”的旧流程，替换成了一个语义上更正确、也已经跑完真实评测的新流程。我们下期再见。
