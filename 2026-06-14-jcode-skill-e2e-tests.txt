欢迎收听本期技术小记。今天的故事,从一个看起来很简单的问题开始:我们到底有没有真正测过那些新功能?

事情是这样的。jcode 这个编码助手,有一套端到端测试,跑起来很硬核:它会真的把守护进程拉起来,通过一行一行的 JSON 消息和它对话,再去检查回来的事件对不对。我们先盘了一遍家底,发现 MCP 这条线测得很扎实,可是技能、还有这次新加的一大批命令行参数,基本是裸奔的状态,没有这种端到端的回归。

于是机主说,那就补上。但他特别提醒了一句很关键的话:手动敲美元符号、弹出技能补全那个弹窗,是界面层的东西,这种无头测试根本照不到,别在这上面较劲。真正要验证的,是三件事:这个技能有没有被加载、有没有被触发、有没有被执行。

这个思路一下子就清晰了。我先去把机制摸透:技能是从工作目录下的点 skills 文件夹里发现的,加载之后会出现在一个叫状态同步的事件里;模型想用它,就会去调一个叫 Skill 的内置工具;工具一跑,就把技能正文原样塞回对话里。三个环节,三个可观测的信号。

然后就是动手写测试、再用真实的模型去跑。跑的过程里,蹦出来好几个有意思的真相。

第一个:Python 后端发出来的技能列表,居然是空的。原来把技能喂进状态同步这个能力,是这次在 Rust 这边才做的,Python 那头还没有。所以加载这条测试,我就专门钉在了 Rust 后端上。

第二个:守护进程刚连上时发的第一条状态同步,会抢在测试订阅之前就溜走了,根本抓不到。解决办法很巧:发一个本地的斜杠 plan 命令,逼它在订阅之后重新广播一次。这一步不花模型、又快又稳。

第三个,也是最有意思的。那个"禁用斜杠命令"的参数,我一开始以为会把斜杠 help 当成普通文字丢给模型。结果真机一跑,它老老实实回了一句"未知命令"。我去翻源码才明白,它用的是一个空的命令分发器,任何斜杠命令都会被判成未知。于是我把测试改成断言它真正的行为,而不是我脑补的行为。

最后,确定性能测准的参数,我都补上了:输出成单个 JSON 结果、追加系统提示词、禁用斜杠命令、还有用参数加载插件让钩子真的触发。而那些没法稳定断言的,比如禁用工具、关闭会话持久化,我没有硬凑,而是写清楚理由,留给单元测试。

六个端到端测试,全部用真实的 Rust 守护进程加真实的模型跑通,格式和静态检查也都干净。

这就是今天的故事:有时候写测试最大的价值,不是它通过了,而是它逼你看清系统真正的样子。我们下期再见。
